# 设计理念：LLM 在本系统中到底是什么角色

> 这是一篇**项目自述型**知识库笔记，记录"为什么这个系统要把 LLM 放在这个位置"。
> 不是外部学来的知识，而是本系统的设计决策——看懂它，就看懂了整套架构的灵魂。

### 1) 一句话核心

LLM 在这里是**"外脑 + 书记员"**——它帮你读文件、归因、结构化、提醒，但**写什么进系统、规则成不成立、系统要不要扩张，全是人说了算**。真正的核心是 markdown 结构 + Plan→Execute→Verify→Reflect→Patch 闭环，LLM 只是让每一步更省力。

### 2) 关键点

**LLM 的 8 个实质用途（全部走同一个咽喉 `assistant.call_llm()`）：**

| 角色 | Prompt 函数 | 干什么 |
|---|---|---|
| 复盘·Verify | `verify_question` | 逐条问你上周 Plan 做到没、没做到的一句话原因 |
| 复盘·Reflect+Patch | `reflect_patch` | 归因为**能力问题 vs 系统问题**（再分 6 类），系统问题提炼 0-1 条规则 |
| 复盘·Plan | `plan` | 给下周 3 个子系统各定 1-2 件，少而精 |
| 踩坑即时 | `pit` | 判断一个坑是"一次性"还是"会重复的系统问题" |
| 沉淀知识 | `learn` | 把任意输入结构化进知识库 |
| 见闻归档 | `archive` | 见闻池条目结构化归档 + 建 `[[双链]]` |
| 灵感分析 | `inspiration` | 把一闪而过的灵感展开成可落地方向 |
| 月度体检 | `checkup` | 判断哪个子系统空转、规则库该不该合并——**默认劝简化** |

（另有 `assistant test` 的连通性测试，验密钥接口用。）

**架构事实：**
- **单一咽喉**：所有调用都过 `call_llm(cfg, messages)`，所以看板里切模型 = CLI 也切，模型管理白送。
- **Prompt 单一来源**：全部集中在 [prompts.py](../../tools/prompts.py)，改 Prompt 只改一处。
- **零依赖**：纯 Python 标准库 urllib 调 OpenAI 兼容接口，有 Python 就能跑。

**LLM 在系统里的定位（4 条设计铁律）：**

1. **LLM 是"省力的手脚"，不是"系统的脑子"。**
   不用它，纯手写复盘一样能跑（见系统说明"用法 A"）。核心是方法论 + 文件结构，AI 只是把每步加速。

2. **人保留最终决定权——LLM 永远不碰规则库正本。**
   提炼出的规则只进 `规则库_候选.md`，**不会自动写进 `规则库.md`**，要复盘确认后手动搬。规则库是系统的"立法权"，不能交给 AI。

3. **LLM 不污染记忆核心。**
   `GBrain启动预案.md` 明确警告：将来接自动记忆系统时，"别让 AI 自动写入污染记忆核心"——几十个 cron 自动改知识库是反模式。

4. **防过度系统化——LLM 是为了省时间。**
   总纲领明说"记录花的时间不能超过它省的时间"。某个 AI 功能让你更累更复杂，就该砍掉。

### 3) 今日意义

- **对使用者**：看懂这条就知道，这个系统不会"被 AI 带着跑"。AI 越聪明，你省的力越多，但**方向盘一直在你手里**。这跟很多"全自动 AI 助手"的设计哲学相反——这里 AI 是工具不是主人。
- **对想 fork 改造的人**：所有 LLM 调用都集中在 `call_llm` 一个函数，想换模型、换 Prompt、加新功能，都只动一处。理解了"候选规则库"这道闸，就知道如何安全地扩展而不破坏系统的可信度。
