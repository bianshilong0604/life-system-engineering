# 个人总体设计部 · 总纲领

> 这不是一个待办清单，是一套**会自己变强的机制**。
> 你是总设计师，负责定目标、协调子系统冲突；执行尽量外包（AI、脚本、工具、别人）。
> 灵感来自钱学森《论系统工程》——把"组织管理"上升为可运行、可反馈、可进化的科学方法。

---

## 怎么用这套东西（30 秒说明）

```
每周日晚 15 分钟 → 打开 02_每周复盘/ 新建一篇 → 跑一遍闭环：
Plan(下周每个子系统定1-2件) → Execute(这周做) → Verify(周末看真实完成度)
→ Reflect(失败的：能力问题 还是 系统问题?) → Patch(系统问题就更新一条规则进 规则库.md)
```

平时不用管它。**只在周日复盘时打开。** 这是它有用的前提——低维护成本。

> 🤖 **不想手动跑？** 打开 [AI复盘助手.md](AI复盘助手.md)，把 Prompt A 整段复制给 Claude Code（或任何支持的工具调用 LLM），它会读你的文件、带你跑完闭环、自动写回复盘和规则库。这就是"人—Agent—Harness 协同进化"。

## 一、我的硬约束（不靠意志力，靠规则锁死）

> 硬约束 = 不商量、不例外、写死。违反它不是"今天没忍住"，是"系统破了，要修"。

- [ ] 23:30 手机进客厅充电（卧室不放手机）
- [ ] 22:00 后 Claude Code / 代码只读不写（防止研究子系统溢出，挤掉睡眠）
- [ ] 每天至少 1 次离开椅子的身体活动（哪怕散步 20 分钟）
- [ ] 重大决定（换方向/大额消费/长期承诺）必须"睡一觉再定"，不当场答应

> *上面是示例约束，第一次复盘时改成你自己的。硬约束的关键是"具体到能判断做到没"。*

## 五、三个核心问题（每次复盘必问）

1. 这次搞砸是**我能力问题**，还是**系统问题**？
2. 是目标不清 / 时间没留 / 工具差 / 边界乱 / 反馈少 / 标准不清——具体哪一类？
3. 有没有**一条规则**能让同类问题下次自动减少？→ 写进 [规则库](规则库.md)

## ⚠️ 防过度系统化（钱学森本人的警告）

- 指标少到**一只手数得过来**，记录花的时间不能超过它省的时间
- 关系、创造力、当下的快乐——**不该被量化**，硬要量化反而毁掉它
- 判断标准：**如果这套东西让你更累更焦虑，就是搭错了，立刻简化**
- 系统是工具，不是目标。它服务于你想过的生活，不是反过来
